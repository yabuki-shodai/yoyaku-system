# Lumi Admin UI ドキュメント

## 目的

このドキュメントは、`apps/admin` に実装されている Lumi 美容院予約管理システムの管理画面 UI 方針をまとめたものです。
画面構成、権限別の表示、共通レイアウト、TailwindCSS の使い方、予約・シフト・メニュー・レポート画面の確認入口として使います。

## デザイン方針

- 美容院スタッフが業務中に使う管理画面として、装飾よりも視認性と操作の速さを優先します。
- UI ラベルは日本語を基本にします。
- 店舗名、スタッフ名、メニュー名などの固有名は英字表記を許容します。
  - 例: `Lumi.`, `RIO`, `MEI`, `SORA`, `HINA`, `TOKIOトリートメント`
- 全体は淡いグレー背景、白い操作面、黒に近いヘッダーで構成します。
- アクセントはスタッフごとの色に限定し、予約カード、シフト状態、集計表示で使います。
- 管理画面ではカードを増やしすぎず、一覧、表、タイムラインなど業務画面として読み取りやすい密度を保ちます。

## 主要画面

| ルート | 役割 |
| --- | --- |
| `/` | `/bookings` へリダイレクト |
| `/login` | スタッフ ID によるログイン |
| `/bookings` | 予約一覧。日別タイムラインと予約カードを表示 |
| `/bookings/new` | 新規予約登録 |
| `/bookings/[id]` | 予約詳細、顧客情報編集、来店完了、キャンセル |
| `/shifts` | スタッフ・シフト管理 |
| `/menus` | メニュー・料金・所要時間の編集 |
| `/reports` | 売上・予約数・人気メニュー・スタッフ別売上の確認 |

## 権限別表示

実装:

- `apps/admin/lib/permission-service.ts`
- `apps/admin/components/layout/ProtectedRoute.tsx`
- `apps/admin/components/layout/AdminShell.tsx`

| ロール | 表示画面 | 主な操作 |
| --- | --- | --- |
| 管理者 | 予約一覧、新規予約登録、スタッフ・シフト、メニュー編集、レポート | 予約作成、予約編集、ステータス更新、全スタッフのシフト編集、メニュー編集 |
| 受付 | 予約一覧、新規予約登録 | 予約作成、予約編集、ステータス更新 |
| スタッフ | 自分の予約、自分のシフト | 自分の予約確認、自分の担当予約の完了、自分のシフト編集 |

ルール:

- 未ログイン時は `/login` へリダイレクトします。
- 権限のない画面へアクセスした場合は `/bookings` へ戻します。
- `/bookings/new` は `booking-detail` 権限の中で扱いますが、作成可否は `canCreateBooking` で判定します。

## 共通レイアウト

### AdminShell

実装:

- `apps/admin/components/layout/AdminShell.tsx`

役割:

- ログイン後の全管理画面を包む共通シェルです。
- 上部に黒背景のヘッダーを表示し、画面遷移用のナビゲーションを横並びにします。
- 現在のルートに応じてアクティブなナビ項目を白背景で強調します。
- 右側にログイン中アカウント、ロール、ログアウトボタンを表示します。

表示項目:

- ブランドロゴ `Lumi.`
- `ADMIN` ラベル
- 権限別ナビゲーション
- アカウント名
- ロール表示
- ログアウト

### ProtectedRoute

実装:

- `apps/admin/components/layout/ProtectedRoute.tsx`

役割:

- ページごとに必要な `screen` を受け取り、ログイン状態と権限を確認します。
- 認可済みの画面だけ `AdminShell` 内に描画します。
- 認可ロジックを page 側へ広げず、各ページの表示責務を軽く保ちます。

## スタイリング方針

定義:

- `apps/admin/app/globals.css`

グローバル CSS:

```css
@import "tailwindcss";
```

ルール:

- グローバル CSS は TailwindCSS の読み込みに限定します。
- 画面固有の見た目は基本的に TailwindCSS のユーティリティクラスで実装します。
- 共通化したい構造は CSS クラスではなく、コンポーネント化を優先します。
- 入力欄、テーブル、カード、ボタンの余白や幅は Tailwind の固定幅、最小幅、grid、flex で制御します。
- 料金・時間入力のように崩れやすい UI は、列幅と入力グループ幅を明示します。
  - 料金列: `w-[180px]`
  - 所要時間列: `w-[150px]`
  - 削除列: `w-[72px]`
  - テーブル最小幅: `min-w-[760px]`

フォント:

- 見出し、数字、ロゴ: `Outfit`
- 本文、日本語 UI: `Zen Kaku Gothic New`

## 状態管理

実装:

- `apps/admin/lib/admin-context.tsx`
- `apps/admin/lib/mock-data.ts`
- `apps/admin/lib/booking-service.ts`
- `apps/admin/lib/auth-service.ts`

役割:

- `AdminProvider` がログインアカウント、予約、メニュー、シフト状態を保持します。
- プロトタイプのため、状態はブラウザの現在セッション内で保持します。
- ページ分割後も Context 経由で共通状態を扱い、URL ごとの画面と業務状態を分離します。

保持する主な状態:

| 状態 | 内容 |
| --- | --- |
| `account` | ログイン中のスタッフアカウント |
| `bookings` | 予約一覧 |
| `menus` | メニューカテゴリとメニュー項目 |
| `shifts` | スタッフ別の週間シフト |
| `shiftSubmitted` | シフト提出済み表示 |

- `apps/admin/app/bookings/[id]/page.tsx`
- `apps/admin/components/bookings/BookingDetail.tsx`

表示:

## スタッフ・シフト UI

実装:

- `apps/admin/app/shifts/page.tsx`
- `apps/admin/components/staff/ShiftManager.tsx`

表示:

- スタッフごとの週間シフト表
- 曜日別の出勤/休み切り替え
- 予約が入っている日のロック表示

レイアウトルール:

## レスポンシブ対応

Header:

- PC: ロゴ、ナビ、アカウントを横並びにします。
- 狭い画面: ヘッダー要素を折り返し、ナビは横スクロール可能にします。

テーブル:

- メニュー編集など横幅が必要な表は `overflow-x-auto` を使います。
- 列幅が崩れないよう `table-fixed` と明示的な `w-[...]` を組み合わせます。

タイムライン:

- スタイリスト列は必要な横幅を維持します。
- 時刻、予約カード、現在時刻ラインが重ならないよう、グリッド内の高さと位置を固定的に扱います。

フォーム:

- 入力欄は `min-w-0` と `w-full` を適切に使い、親要素からはみ出さないようにします。
- ボタンの文言が折り返しても操作面が崩れないよう、最小高さを設定します。

## 実装時の注意

- UI 文言の汎用ラベルは日本語を基本にします。
- グローバル CSS に画面固有スタイルを追加しません。
- まず TailwindCSS のユーティリティクラスで表現し、重複が増えたらコンポーネント化します。
- ページごとの認可は `ProtectedRoute` に寄せます。
- 権限判定は `permission-service.ts` に集約します。
- 予約やメニューの計算処理は `components` に入れず、`lib` 側へ置きます。
- ページ分割後の遷移は URL を正とし、画面状態だけでページを切り替えないようにします。
- テーブル、カード、フォームの幅はレスポンシブ時の崩れを確認してから変更します。
