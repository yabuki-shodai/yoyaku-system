# 03. 非機能要件

## 3.1 性能
- 予約空き状況の検索は、一般的な操作で体感1〜2秒以内に結果を返すこと。
- 同時アクセス数は小規模店舗を想定し、当面は数十〜百人規模の同時利用に耐えられればよい。

## 3.2 可用性
- 店舗の営業時間帯を中心に高い可用性を確保する。計画メンテナンスは営業時間外に実施する。

## 3.3 セキュリティ
- パスワードはハッシュ化（bcrypt等）して保存する。
- 認証トークン（JWT等）は有効期限を設け、失効・再発行の仕組みを持つ。
- 一般ユーザーは他ユーザーの予約・個人情報にアクセスできないよう認可制御を行う。
- 管理画面（apps/admin）は管理者ロール以外からアクセスできないようにする。
- 個人情報（氏名・電話番号・メールアドレス等）は適切にアクセス制御し、ログ等に平文で残さない。

## 3.4 データ整合性
- 二重予約（同一スタッフ・同一時間帯への重複予約）が発生しないよう、予約確定時に排他制御を行う。
- 予約のキャンセル・変更履歴を追跡できるようにする（監査目的、詳細は [[06_open_questions]]）。

## 3.5 運用・保守性
- 管理者はメニュー・スタッフ・営業時間などのマスタデータをコードデプロイなしで変更できること。
- ログ（アプリケーションログ・エラーログ）を収集し、障害調査に利用できること。

## 3.6 拡張性
- 将来的な複数店舗展開、オンライン決済連携、LINE通知連携等を見据え、店舗・通知手段をハードコードしない設計とすることが望ましい（本フェーズでは実装しないが、データモデル上大きな手戻りが出ないよう配慮する）。

## 3.7 アクセシビリティ・対応環境
- 一般ユーザー向け画面はスマートフォンでの利用を主とし、レスポンシブ対応必須。
- 主要ブラウザ（Chrome, Safari, Edge）の最新版で動作すること。

## 3.8 多言語対応
- 本フェーズは日本語のみ対応とする。
